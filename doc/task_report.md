Built the core analysis module for Script Pulse. I started by setting up the actual spine of the project instead of leaving it as a vague idea with scattered prompts. That meant defining clear output shapes for every major stage of the pipeline, from raw script input and beat extraction to emotional analysis, engagement scoring, critique suggestions, final report assembly, and validation. I also wrote the prompt templates that drive each stage so the whole thing had one shared language instead of every step drifting off in its own direction. Part of that cleanup was renaming the main module from a generic source folder into a proper core module so the structure matched what the project had actually become.

I created the first pipeline flow document so the sequence of the system was no longer trapped in my head. I mapped the analysis into a short step-by-step path from input to final output, keeping it simple enough to scan quickly but concrete enough that anyone opening the repo could understand how the tool moves through the script. That doc became the backbone for later changes because once the flow was written down, it was easier to spot where the design was bloated or confused.

I then built a shared per-stage validation layer and wired the flow around retry loops at each step. At that point the idea was to have every stage check its own work before the next stage ran, which sounds nice on paper and honestly looked disciplined at first. I added a reusable validation structure and a shared validation prompt so the same logic could be reused across stages instead of copy-pasting checks everywhere. It was real work, not hand-wavy theory, and it gave the system a proper first pass at self-correction.

After pushing on that design a bit more, I realized the per-stage validation route was becoming too fussy and too expensive for what it gave back. So I ripped that version out and replaced it with a regeneration loop that happens after the final validation, which is a lot cleaner because it judges the whole report instead of nitpicking every stage in isolation. I updated the flow so the retry story made sense at the full-run level, not the micro level. That was one of those annoying but necessary reversals where the first idea was not terrible, just not the right shape for the product.

I did not stop there, because even the separate regeneration layer felt like one abstraction too many. I folded the regeneration instructions directly into the final validator so the validator itself could say whether the output was retryable and how the next pass should be guided. That cut out extra moving parts and made the final loop easier to reason about. In plain English, I made the system judge its own output and also explain how to fix it, all in one place, instead of splitting that responsibility across disconnected pieces.

I cleaned up the prompt design so the hard rules and output shapes lived in the system prompts and the user prompts stopped carrying too much weight. Before that, the templates were mixing instructions, formatting rules, and input content in ways that were harder to maintain and easier to break. I moved the structural expectations into the part of the prompt that should be stable and left the user-facing input layer much lighter. That sounds subtle, but it matters a lot because it reduces prompt clutter and makes stage behavior more consistent.

I added a context builder that assembles the messages for each stage in a more deliberate order. Instead of stuffing everything into one giant prompt blob, I split the incoming material into separate context messages and positioned the stable instruction layer near the end so it stayed close to the actual request. This was a practical prompt engineering fix, not cosmetic cleanup. It made the stages easier to compose, easier to debug, and less likely to get weird because of prompt bloat or truncation.

I also built a safer serialization utility for prompt inputs so structured data could be passed through cleanly without turning every message into a messy wall of text. When the input was a dictionary-like data block, I compacted it in a predictable way. When it was just plain text or a list, I left it readable instead of forcing fake JSON everywhere. That gave the prompt layer a much cleaner interface and removed a whole class of small, stupid formatting failures that waste time for no reason.

Once the structure was stable enough, I polished the tone of the prompts themselves. I added stronger persona lines to the system prompts so the analysis felt more focused and less generic, and I rewrote the user prompt templates to sound more human. This was not just style vanity. If the prompt voice is stiff and robotic, the output usually follows it into the same ditch. I tightened that up so the system could feel sharper without sounding like template sludge.

I then built the input normalization module, which is one of those unglamorous pieces that quietly decides whether the rest of the pipeline works or falls apart. I made it trim and line-number script input, detect likely character names, infer the general format of the script, and keep a line map so later analysis could point back to exact evidence in the source. I also refined the pattern matching so scene headings and parenthetical dialogue names were recognized properly while fake character labels like titles and scene labels were ignored. That made the input much cleaner and gave the later stages something reliable to stand on.

I did manual testing for the script normalizer with a bunch of ugly edge cases because this kind of thing always breaks on the weird examples, not the clean ones. I tested empty input, tabs, multi-word names, numeric names, lowercase scene headings, non-dialogue lines that only look like dialogue, the assignment example, and random unstructured text. I also fixed the test runner so it could be run directly without path drama, and I cleaned the output so it logged properly instead of spraying raw prints. After the regex fixes, the manual test sweep passed cleanly.

I implemented the full large language model analysis loop and connected it to UltraGPT so the whole pipeline could actually run end to end. This was where the project stopped being mostly structure and started behaving like a product. I built the analysis runner, the client wrapper around the model gateway, the dependency file, environment key loading, regeneration context support, and schema-safe parsing. I also made the loop accept prior reports and custom regeneration guidance so later runs could improve based on earlier failures instead of starting from zero every time.

I added proper logging and error handling because once the pipeline became real, silent failures would have been unforgivable. I made the run log iteration stats, stage-level token usage, validation results, retry counts, and failures in a way that could actually be read later. I also cleaned up the logging style so errors were reported consistently instead of with a mix of noisy patterns. It was one of those boring but necessary passes that saves hours later when something goes sideways.

I explored loosening the engagement factor schema to allow custom scoring categories beyond the fixed rubric. That gave the system a temporary path for handling extra factors with custom names, but after looking at the tradeoff, I rolled it back conceptually and kept the scoring strict unless explicitly needed. That was the right call because letting scoring dimensions drift is a clean way to destroy consistency while pretending you are adding flexibility. I spent time there so I knew exactly why it should stay tighter.

I then built the FastAPI backend and SSE-based web interface with SQLite session tracking so Script Pulse could work like an actual app instead of a command-line experiment. The backend could create sessions, store script input and config, stream live progress during analysis, handle regeneration prompts, and persist results. On the frontend side, I built a single-page interface with session history, live run updates, report views, config controls, and database-backed state so a user could leave and come back without losing context. I also moved database storage into a local temp folder, added proper app startup handling, allowed sessions to be deleted, and tightened the run flow so draft sessions behaved differently from completed ones.

The earlier version functioned, but it still had that obvious first-pass energy where the structure is there but the feel is off. That pass mattered because good tools die all the time under bad presentation.

I spent real time separating model judgment from deterministic math in the engagement system, because that line matters more than people think. A language model can judge whether something feels strong, weak, tense, clear, or flat. It should not be trusted to do the final arithmetic that the rest of the system relies on. I worked through the scoring logic to pin down exactly what should remain subjective and what should be recomputed in code every single time so the numbers stop drifting.

I then implemented deterministic engagement factor weights and moved the actual weighted-score and overall-score math into code. This fixed the stupid 6 versus 7 mismatch problem and made the scoring consistent regardless of what the model tried to output numerically. In practice, the model now provides the qualitative scoring judgments, and the code overwrites the arithmetic using fixed weights that sum cleanly. That is the right division of labor, and it removed a whole category of flaky validation problems.

I aligned the rubric with the schema, made the pipeline load weights from the rubric instead of hard-coding them elsewhere, validated that the factor set matched exactly, and recomputed the weighted totals from that shared source of truth. That made the scoring system coherent again instead of stitched together.
